# Verification tools

Three complementary tools verify that compiled nftables rulesets
behave the same as the iptables baseline they were migrated from.
Each catches a different class of bug.

| Tool | Checks | Speed | Needs |
|------|--------|-------|-------|
| Triangle verifier | Rule equivalence (static) | seconds | iptables-save dump |
| Packet simulator | Verdict equivalence (dynamic) | minutes | iptables-save + 3 netns + scapy |
| Connection-state tests | Stateful rule correctness | seconds per scenario | scapy |

## Triangle verifier

`shorewall-nft verify` compares the rules generated by the
shorewall-nft compiler against a captured `iptables-save` dump from
a running Shorewall host. It's called the triangle because it
compares three things:

```
    Shorewall config (.../etc/shorewall)
           │
           │ compiles to
           ▼
    nft ruleset (what shorewall-nft emits)
           │
           │ compared to
           ▼
    iptables dump (what legacy Shorewall produced)
```

The comparison is semantic: rules are normalized into fingerprints
so minor syntactic variations don't matter. The output is a
per-zone-pair coverage report.

### Usage

```bash
# IPv4 + IPv6 automatically (looks for ip6tables dump next to iptables)
shorewall-nft verify /etc/shorewall --iptables /tmp/iptables-save.txt

# The ip6tables dump is auto-detected by name substitution:
# /tmp/iptables-save.txt → /tmp/ip6tables-save.txt
```

Capture the dumps from the legacy Shorewall host first:

```bash
# On the legacy host:
sudo iptables-save --counters > /tmp/iptables-save.txt
sudo ip6tables-save --counters > /tmp/ip6tables-save.txt
```

### Output

```
IPv4: [PASS] 100.0% coverage (8281/8281) | 240/241 pairs | 0 order-conflicts
IPv6: [PASS] 100.0% coverage (3310/3310) | 60/60 pairs | 0 order-conflicts
```

- **`coverage`** — rules in iptables that have a matching rule in
  our nft output
- **`pairs`** — zone-pair chains fully matched
- **`order-conflicts`** — rules in different order, possible
  first-match semantics change

Non-100% coverage lists the missing rules. Investigate each one by
compiling with `-o`:

```bash
shorewall-nft compile /etc/shorewall -o /tmp/compiled.nft
# Compare /tmp/compiled.nft against /tmp/iptables-save.txt manually
```

### Production verification workflow

Full regression on multiple production firewalls:

```bash
for fw in /srv/fw-configs/*/shorewall; do
    name=$(basename $(dirname "$fw"))
    shorewall-nft verify "$fw" \
        --iptables /srv/fw-dumps/iptables-$name.txt
done
```

Target: **100.0% coverage on every config**. Anything less is a
compiler regression and blocks release.

### Running as a test

```bash
pytest tests/test_triangle.py -v
```

The triangle tests gate on `PROD_DIR.exists()` — they skip if
there's no production config to compare against.

## Packet simulator

Static verification can miss bugs in rule ordering or conntrack
handling. The packet simulator loads the compiled nft ruleset in a
middle namespace and sends real TCP/UDP/ICMP packets through it
from a source namespace to a destination namespace:

```
 shorewall-next-sim-src  ←veth→  shorewall-next-sim-fw  ←veth→  shorewall-next-sim-dst
      (source host)              (firewall under test)           (destination host)
```

Packets are crafted with scapy and verdicts are observed by:

- Checking whether the packet arrives at the destination
- Watching the `nft monitor trace` stream in parallel
- Querying per-rule counters

Each test case consists of (source IP, dest IP, protocol, port)
and an expected verdict from the iptables dump.

### Usage

```bash
shorewall-nft simulate /etc/shorewall \
    --iptables /tmp/iptables-save.txt \
    --target 203.0.113.5 \
    --max-tests 60 \
    --seed 42 \
    -V            # verbose: show all test results, not just failures
```

Options:

- `--target IP` — focus tests on packets involving this IP
- `--targets LIST|@FILE` — comma-separated list or `@file` of target
  IPs. All targets share a single netns topology so the nft ruleset
  is loaded exactly once regardless of how many targets you pass.
  This avoids the CPU burst from reloading a multi-thousand-rule
  ruleset per target.
- `--ip6tables FILE` — ip6tables-save dump for IPv6 tests
- `--targets6 LIST|@FILE` — IPv6 target addresses (requires `--ip6tables`)
- `--src-iface NAME` — override the src-zone interface name in the
  FW netns (default: `bond1`, net zone)
- `--dst-iface NAME` — override the dst-zone interface name in the
  FW netns (default: `bond0.20`, host zone)
- `--max-tests N` — cap the number of random test cases per target (default 60)
- `--seed N` — reproducible sampling
- `--parallel N`, `-j N` — parallel workers (default 4)
- `--no-trace` — disable nft trace capture (faster but less diagnostic)

### Topology and coverage

The simulator sets up three network namespaces with two dual-stack
veth pairs:

```
NS_SRC                     NS_FW                              NS_DST
(src-z veth)               (src_iface + dst_iface)            (dst-z veth)
10.200.1.2/30              10.200.1.1/30  ↔  10.200.2.1/30    10.200.2.2/30
fd00:200:1::2/64           fd00:200:1::1  ↔  fd00:200:2::1    fd00:200:2::2/64
+spoofed saddr (v4/v6)     SRC_IFACE=bond1   DST_IFACE=bond0.20 +target IPs
                           (net zone)        (host zone)
```

By default this exercises **`net → host` (FORWARD)** only, both v4
and v6 if an `--ip6tables` dump is supplied. To test other zone
pairs, override the interface names to match interfaces from other
zones in your config:

```bash
shorewall-nft simulate /etc/shorewall46 \
    --iptables iptables-save.txt \
    --src-iface bond0.18 \   # adm zone
    --dst-iface bond0.20 \   # host zone
    --target 203.0.113.130
```

Not currently exercised by simulate:

- INPUT / OUTPUT chains (firewall itself as src or dst)
- Reverse traffic (host→net)
- Intra-zone rules (net→net)
- Rule chains for zones whose interface isn't `src_iface` or
  `dst_iface` — use `verify --iptables` for full static rule-level
  coverage, which has no topology constraint and handles all 241
  zone pairs via fingerprint comparison.

### Failure investigation

A failure looks like:

```
[015/060] TCP 10.0.0.5:54321 → 203.0.113.5:80 expect=ACCEPT got=DROP ✗

  iptables expected: ACCEPT (matched rule "ACCEPT loc web tcp 80" at iptables-save.txt:142)
  nft observed:      DROP (matched rule at compiled.nft:873)
                     "meta nfproto ipv4 ip saddr 10.0.0.0/8 drop"
  trace excerpt:
    trace id ... inet shorewall forward rule jump loc-web (verdict jump loc-web)
    trace id ... inet shorewall loc-web rule meta nfproto ipv4 ip saddr 10.0.0.0/8 drop
```

The failing trace excerpt is saved to
`/tmp/shorewall-next-sim-trace.log`.

### Requirements

- `scapy` installed: `pip install -e ".[simulate]"`
- `run-netns` tooling installed (see [Setup](setup.md))
- Sufficient capabilities: the test user must be in the
  `netns-test` group

## Connection-state tests

The connstate module verifies that stateful rules handle the full
TCP lifecycle and edge cases correctly:

- 3-way handshake
- SYN + FIN (invalid)
- SYN + RST (invalid)
- Out-of-state ACKs
- Invalid flag combinations

Each test crafts a packet sequence in scapy and asserts the expected
verdict against the loaded ruleset.

```bash
# Run the connstate suite against the default config
pytest shorewall_nft/verify/connstate.py -v
```

The module is also importable for custom scenarios:

```python
from shorewall_nft.verify.connstate import run_connstate_tests

results = run_connstate_tests(
    config_dir="/etc/shorewall",
    target_ip="203.0.113.5",
)
for r in results:
    print(f"{r.name}: {'PASS' if r.passed else 'FAIL'}")
```

## Combining verification tools

A **full regression run** uses all three:

```bash
#!/bin/bash
set -euo pipefail

CONFIG=/etc/shorewall
IPT_DUMP=/tmp/iptables-save.txt

# 1. Static check (fast, 100% required)
shorewall-nft verify "$CONFIG" --iptables "$IPT_DUMP" || exit 1

# 2. Packet simulator (slower, 100% required)
shorewall-nft simulate "$CONFIG" \
    --iptables "$IPT_DUMP" \
    --max-tests 100 -j 4 || exit 1

# 3. Connection state (edge cases)
pytest shorewall_nft/verify/connstate.py -v || exit 1

echo "All verification passed ✓"
```

Wire this into CI after every merge.
